# NexusAPI Evaluator 真实流程验收清单与试用测试方案

本文档用于真实 API 试用验收。目标不是证明“代码能跑”，而是判断这个工具是否已经适合交给非技术测试人员按流程完成模型/渠道评测。

## 1. 验收目标

本轮验收需要确认：

- 新用户能按页面引导完成 API 配置、标准评测、报告中心查看和测试交付。
- 新用户能使用配置模板减少协议选择错误。
- 保存 API 配置前能发现明显错误，并用简单话术提示操作员。
- 测试人员能填写项目、批次、测试人员和测试目的，并在交付模板里看到这些信息。
- 非技术人员能使用“标准评测”完成一次低成本初筛。
- 测试前能看到请求数和 token 消耗预估。
- 测试中能看到进度，并能取消长任务。
- 测试后能得到可读的 Markdown / HTML 报告。
- 报告能给出成功率、P95、错误分布、错误诊断和下一步建议。
- 保存并测试配置、快速测试或标准评测失败时，页面能给出可执行的错误处理建议。
- 标准评测完成后，页面能给出人话结论和下一步按钮。
- 内部错误只向用户展示简单提示和错误编号，技术细节写入本地 `NexusAPI数据/日志/errors.jsonl`。
- 报告中心能显示“极简结论”，非技术人员不用先理解技术指标。
- 操作员能一键导出脱敏问题包给负责人排查。
- 报告中心能生成模型/渠道排行榜，帮助快速横向比较。
- 测试交付页能生成可直接发给负责人的交付模板。
- API Key 不会写入 `profiles.json`、日志、报告或导出配置。
- 程序异常关闭后，任务状态能标记为“已中断”。

## 2. 验收前准备

### 2.1 环境准备

在项目目录执行：

```bash
pnpm dev:desktop
```

如果只跑 Web MVP，可分别执行：

```bash
pnpm dev:server
pnpm dev
```

注意：

- 不要把工具端口设置成 VPN 或代理端口。
- 当前默认端口是 `5179` 和 `5180`。
- 如果端口冲突，按启动提示换端口。

### 2.2 API 准备

至少准备 2 个真实被测配置：

| 配置 | 用途 | 要求 |
|---|---|---|
| API-A | 基准渠道 | 你认为相对稳定的渠道 |
| API-B | 对比渠道 | 另一个模型、渠道或中转线路 |

建议额外准备 1 个故意错误配置：

| 配置 | 用途 | 示例 |
|---|---|---|
| API-Bad | 错误诊断测试 | 错模型名、错 Key、错协议任选一种 |

### 2.3 成本控制

首次验收不要直接跑高成本测试。

推荐顺序：

1. 标准评测：每个 API 先跑一次初筛。
2. 快速测试：错误配置或需要排查时单独跑。
3. 报告中心：先看极简结论、排行榜和报告路径。
4. 测试交付：复制交付模板给负责人。
5. 稳定性冒烟：负责人要求更稳妥时，每个 API 3 轮。
6. 稳定性基础：候选 API 10 轮。
7. 场景测试：需要补充能力表现时，先选 2-3 个场景。
8. 批量测试：需要横向对比时，先 2 个 API，每个 3 轮。

## 3. 验收流程

### 3.1 启动与首页引导

操作：

1. 启动工具。
2. 进入“总览”页。
3. 填写本次测试信息。
4. 查看“推荐操作路径”和“当前建议”。

验收点：

- 页面能正常打开。
- 左侧导航完整显示。
- 总览页显示 API 配置数量、最近请求数量、测试报告数量。
- 本次测试信息能保存，刷新页面后仍能看到。
- 没有配置时，“当前建议”应提示先配置 API。
- 点击“当前建议”按钮能跳到正确页面。

记录：

```text
启动方式：
是否正常打开：
本次测试信息是否保存：
当前建议是否准确：
发现的问题：
```

### 3.2 API 配置与 Key 安全

操作：

1. 进入“API 配置”。
2. 使用“AI 中转站 / OpenAI 兼容”配置模板新增 API-A。
3. 使用合适模板新增 API-B。
4. 新增 API-Bad。
5. 查看已配置 API 列表。
6. 使用“更新 Key”修改其中一个 API 的 Key。
7. 导出配置。
8. 尝试把 Base URL 填成 `https://api.example.com/v1/chat/completions`，确认工具阻止保存。
9. 尝试把超时改成 `10000`，确认工具给出风险提示但允许确认保存。
10. 使用“保存并测试配置”保存 API-A，确认自动跳到快速测试结果。

验收点：

- 必填字段为空时有提示。
- 配置模板能自动设置供应商、协议、超时、备注和输入提示。
- 明显错误的 Base URL 会被阻止保存。
- 协议、超时、最大输出等风险会被提示。
- “保存并测试配置”能自动完成保存和快速测试。
- 保存后列表能显示配置。
- 列表不显示真实 Key，只显示已安全保存。
- “更新 Key”能单独更新密钥。
- 导出的 JSON 不包含真实 API Key。
- `NexusAPI数据/配置/profiles.json` 不包含真实 API Key。

手动检查：

```bash
rg "你的真实Key片段" NexusAPI数据/配置/profiles.json NexusAPI数据/日志/requests.jsonl NexusAPI数据/日志/test-runs.jsonl NexusAPI数据/日志/task-events.jsonl NexusAPI数据/报告
```

预期：

- 不应该搜到真实 Key。

记录：

```text
API-A 名称：
API-B 名称：
API-Bad 错误类型：
导出配置是否不含 Key：
profiles.json 是否不含 Key：
发现的问题：
```

### 3.3 标准评测

操作：

1. 进入“标准评测”。
2. 选择 API-A。
3. 稳定性轮数保持 3 轮。
4. 点击“开始标准评测”。
5. 等待三步流程完成。
6. 对 API-B 重复一次。

验收点：

- 标准评测页面能说明三步流程。
- 快速测试失败时应停止后续步骤，避免继续消耗 token。
- 快速测试成功后，应继续执行稳定性测试。
- 稳定性测试完成后，应继续执行少量场景测试。
- 每一步都应显示运行中、完成或失败状态。
- 标准评测结果应给出快速测试、稳定性、复杂场景和下一步建议。
- 如果标准评测失败，结果区应显示错误处理建议。
- 标准评测完成后应显示人话结论。
- 下一步按钮应优先支持跳到交付页，也能按需跳到稳定性测试、场景测试或报告中心。
- 报告中心应能看到新生成的稳定性报告、场景报告和排行榜数据。

记录：

```text
API-A 标准评测是否完成：
API-A 标准评测结论：
API-B 标准评测是否完成：
API-B 标准评测结论：
失败步骤：
是否节省了手动操作：
发现的问题：
```

### 3.4 快速测试

操作：

1. 进入“快速测试”。
2. 选择 API-A，运行默认 Prompt。
3. 选择 API-B，运行默认 Prompt。
4. 选择 API-Bad，运行默认 Prompt。

验收点：

- 正确配置应返回成功。
- 错误配置应返回失败。
- 失败时应显示标准错误。
- 失败时应显示错误处理建议，而不是只显示一行报错。
- 最近请求记录应增加。
- 如果是错 Key，应归类为 `auth_failed`。
- 如果是错模型，应归类为 `model_not_found`。
- 如果是协议不匹配，应尽量归类为 `empty_response` 或 `invalid_response`。

记录：

```text
API-A 快速测试结果：
API-B 快速测试结果：
API-Bad 快速测试结果：
错误分类是否符合预期：
发现的问题：
```

### 3.5 稳定性测试

操作：

1. 进入“稳定性测试”。
2. 对 API-A 跑 3 轮冒烟测试。
3. 对 API-A 跑 10 轮基础稳定性。
4. 对 API-B 跑 3 轮冒烟测试。

验收点：

- 执行前预估显示请求数和 token 消耗范围。
- 点击开始前有确认弹窗。
- 测试中显示进度条。
- 完成后显示汇总结论。
- 生成 Markdown 报告。
- `NexusAPI数据/报告/` 下生成 `.md` 和 `.html`。
- 报告包含成功率、平均耗时、P50、P95、错误分布、诊断建议。
- “最近任务状态”显示已完成。

记录：

```text
API-A 3轮成功率：
API-A 10轮成功率：
API-A P95：
API-B 3轮成功率：
报告路径：
报告结论是否可读：
发现的问题：
```

### 3.6 批量并发测试

操作：

1. 在稳定性测试页找到“批量并发测试”。
2. 选择 API-A 和 API-B。
3. 使用“批量快速筛查：每个 API 3 轮”。
4. 同时测试 API 数保持 2。
5. 单 API 请求并发保持 1。

验收点：

- 多选 API 正常。
- 执行前预估能反映 API 数量。
- 报告能列出多个 API 的成功率、平均耗时、P95、建议。
- 批量结论能给出优先候选。
- 如果某个 API 明显失败，报告里应能看出来。

记录：

```text
参与批量测试的 API：
批量测试总请求数：
优先候选：
失败或波动配置：
发现的问题：
```

### 3.7 场景测试

操作：

1. 进入“场景测试”。
2. 选择 API-A。
3. 使用“低成本初筛包”模板。
4. 确认工具自动勾选连通、速度、结构化等低成本场景。
5. 重复次数保持 1。
6. 如果结果正常，再切换“编程能力包”和“长上下文包”，确认自动选择的场景合理。
7. 最后选择 API-A 和 API-B，跑“基础全量包”1 次。

验收点：

- 场景多选正常。
- 场景包能自动选择对应场景并更新说明。
- 低成本初筛包不会消耗过高 token。
- 结果报告包含每个场景的成功率、质量分、耗时、问题摘要。
- JSON 场景能检查是否输出可解析 JSON。
- 编程、长上下文、推理场景能体现复杂任务差异。
- 报告里明确说明质量分只是规则化筛查。

记录：

```text
小范围场景：
API-A 平均质量分：
API-B 平均质量分：
低分场景：
人工抽查是否认可评分：
发现的问题：
```

### 3.8 取消与中断恢复

取消测试：

1. 启动一个 10 轮或 30 轮测试。
2. 在运行中点击“取消当前任务”。
3. 等待当前请求结束。

验收点：

- 页面显示已请求取消。
- 最近任务状态显示已取消。
- 已取消任务不会被当作完整结论。

中断测试：

1. 启动一个较长的场景测试。
2. 测试运行中关闭程序窗口或终端。
3. 重新启动工具。
4. 进入“报告中心”查看最近任务状态。

验收点：

- 上次未完成任务应显示“已中断”。
- 测试交付模板应列出中断任务。
- 中断任务不应被当成有效报告。

记录：

```text
取消任务是否正常：
中断后是否显示已中断：
交付模板是否列出中断任务：
发现的问题：
```

### 3.9 报告中心、排行榜与交付

操作：

1. 进入“报告中心”。
2. 查看“极简结论”。
3. 查看最新结论卡片。
4. 查看模型/渠道排行榜。
5. 点击“导出问题包”。
6. 查看最近测试报告、最近任务状态、最近请求记录。
7. 进入“测试交付”。
8. 勾选交付前检查清单。
9. 点击复制交付模板。

验收点：

- 最新结论卡片能看懂。
- 极简结论能给出推荐、观察或不推荐，并说明原因和下一步。
- 排行榜能显示 API-A/API-B 的综合分、成功率、P95、质量分。
- 排行榜排序应大致符合真实测试结果。
- 排行榜文案应明确它只是快速筛选，不是最终采购结论。
- 任务状态能看懂。
- 测试交付页能显示项目、批次、测试人员和测试目的。
- 交付模板包含总体结论、最近请求、异常记录、报告附件、负责人确认项。
- 交付模板包含项目、批次、测试人员和测试目的。
- 交付内容不包含 API Key。
- 问题包能下载，且不包含 API Key、完整请求正文或大段原始响应。
- 非技术人员只看交付模板也能知道当前结果。

记录：

```text
交付模板是否可直接发送：
排行榜是否有参考价值：
是否需要手动补充信息：
报告路径是否正确：
发现的问题：
```

## 4. 验收通过标准

本轮验收通过需要满足：

- API 配置、标准评测、报告中心、测试交付都能完成。
- 高级复测入口里的快速测试、稳定性测试、批量测试、场景测试都能完成。
- 标准评测能完成一次完整初筛，并能在失败时停止后续步骤。
- 报告中心排行榜能正确聚合稳定性和场景测试结果。
- 至少 1 个真实 API 能生成稳定性报告和场景报告。
- 报告结论能帮助判断“继续测试 / 观察 / 暂不建议”。
- 错误配置能产生可理解的错误分类。
- API Key 不出现在导出配置、日志、报告、交付模板中。
- 取消和中断任务能被识别。
- 非技术人员按手册和页面引导能完成一次基础测试。
- 错误日志存在，且不会向普通用户暴露堆栈、API Key 或内部接口细节。

## 6. 最小试用组合

如果时间有限，至少跑下面这组：

1. API-A 保存并测试配置 1 次。
2. API-A 标准评测 1 次。
3. 查看报告中心极简结论和排行榜。
4. 测试交付模板复制一次。
5. API-A 稳定性 10 轮。
6. API-A 场景测试低成本初筛包。
7. API-Bad 快速测试 1 次。
8. 检查 `profiles.json`、日志、报告和交付模板中是否没有真实 Key。

这组跑通后，说明工具主流程已经具备试用价值。
