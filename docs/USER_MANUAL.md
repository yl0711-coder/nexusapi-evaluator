# NexusAPI Evaluator 使用手册

本文档面向非技术测试人员和协作测试人员，目标是让使用者知道：每个功能什么时候用、怎么用、大概会消耗多少额度、测试结果怎么看、报告在哪里。

如果要做完整真实流程验收，请使用：

```text
docs/ACCEPTANCE_TEST_PLAN.md
```

## 1. 工具定位

NexusAPI Evaluator 是一个本地模型与渠道评测工具，用来测试 AI API、模型渠道或中转服务的可用性。

它主要回答这些问题：

- API 配置是否能正常连通？
- URL、Key、模型名、协议是否配置正确？
- 某个模型/渠道响应速度快不快？
- 多轮请求是否稳定？
- 是否有超时、限流、上游 5xx、空响应等问题？
- 多个模型/渠道横向对比，哪个更值得继续测试？
- 复杂任务下，模型是否能完成编程、长文本、JSON、推理、写作等任务？

## 2. 基本使用流程

推荐按顺序使用，不要一开始就跑最贵、最复杂的测试。

1. 进入“API 配置”，新增一个被测 API。
2. 回到“总览”，填写本次测试信息，包括项目、批次、测试人员和测试目的。
3. 如果不熟悉工具，进入“标准评测”，选择一个 API 后一键跑完整初筛。
4. 如果标准评测失败，或要手动排查，进入“快速测试”，确认配置能跑通。
5. 如果负责人要求更稳妥，进入“稳定性测试”，先跑 3 轮冒烟测试。
6. 如果 3 轮没问题，再按需要跑 10 轮基础稳定性。
7. 如果要对比多个模型/渠道，使用“批量并发测试”。
8. 如果要测试复杂任务能力，使用“场景测试”。
9. 只有重点候选模型/渠道，才跑 30 轮较完整稳定性。
10. 每次开始测试前先看“执行前预估”，确认这次测试大概会消耗多少额度。
11. 长任务运行时可以看进度条，必要时点击“取消当前任务”。
12. 测试完成后进入“报告中心”，先看结论卡片和排行榜，再看日志。
13. 进入“测试交付”，复制交付模板，把结论、异常和报告路径发给负责人。

## 3. 页面和功能总览

### 3.1 总览

总览页用于快速查看当前工具状态。

显示内容：

- API 配置数量
- 最近请求数量
- 测试报告数量
- 本次测试信息表单
- 给非技术操作员、负责人、交付人员看的说明卡片
- 推荐操作路径
- 当前建议

什么时候看：

- 打开工具后先看一眼，确认是否已经有配置和历史记录。
- 测试后看报告数量是否增加。
- 不确定下一步做什么时，按“当前建议”按钮继续。

推荐操作路径：

| 步骤 | 目的 | 操作 |
|---:|---|---|
| 1 | 配置 API | 填写 URL、Key、模型名和协议 |
| 2 | 标准评测 | 一键执行快速测试、3 轮稳定性和少量场景 |
| 3 | 报告中心 | 先看极简结论、排行榜、测试记录和请求日志 |
| 4 | 交付结论 | 复制交付模板，说明结论、异常、附件和下一步 |

高级测试什么时候用：

- `快速测试`、`稳定性测试`、`场景测试` 是复测和排查入口，不是普通操作员的默认主流程。
- 标准评测不够、负责人要求扩大轮数、需要多渠道横向比较，或需要单独测编程/长文本/内容安全时，再进入高级测试。

当前建议：

- 工具会根据已有配置、请求记录和测试报告，提示下一步应该做什么。
- 如果还没有 API 配置，会建议先去“API 配置”。
- 如果已有配置但还没有测试报告，会建议先去“标准评测”。
- 如果已有测试报告，会建议去“测试交付”复制交付模板。

本次测试信息：

| 字段 | 怎么填 | 用途 |
|---|---|---|
| 项目 / 客户 | 例如 `NexusAPI 渠道复测` | 说明这次测试属于哪个项目 |
| 测试批次 | 例如 `2026-05-24 第一批` | 方便多轮测试对比 |
| 测试人员 | 例如 `测试人员 A` | 交付时说明是谁执行 |
| 测试目的 | 例如 `筛选稳定候选渠道` | 让负责人快速理解测试背景 |

这些信息只保存在本机浏览器 localStorage，会自动写入“测试交付”的 Markdown 模板。

页面内短帮助：

- 每个页面顶部都有“这个页面怎么用？”按钮。
- 点开后会显示 3 条左右的当前页面操作提醒。
- 新手不需要先读完整手册，可以先按页面短帮助执行。

演示模式：

- 总览页有“查看演示数据”按钮。
- 演示数据只用于熟悉界面，不会发起真实请求，也不会写入真实配置。
- 演示模式下可以看到缺 Key 提示、失败记录、排行榜和报告结论的样子。
- 演示完点击“退出演示模式”，工具会恢复本机真实数据。

### 3.2 API 配置

API 配置页用于保存被测模型/渠道的连接信息。

页面上方有三张提示卡：

- Base URL：说明基础地址应该怎么填。
- 协议：说明大多数中转站优先选择 OpenAI Compatible。
- API Key：说明 Key 只保存在本机安全存储，导出配置和报告都不会携带真实 Key。

字段说明：

普通操作员默认只需要填写“必填信息”区域：

- 配置模板
- 名称
- Base URL
- 默认模型
- API Key

“更多参数”默认折叠，只有负责人明确要求时再展开。这里面包含角色、供应商、协议、渠道标识、最大输出、超时毫秒和备注。

| 字段 | 怎么填 | 说明 |
|---|---|---|
| 配置模板 | AI 中转站 / Claude 原生 / OpenAI 官方 / DeepSeek | 不懂协议时先用模板，模板会自动设置协议、供应商、超时和提示 |
| 角色 | 被测 API / 主 API | 当前主要使用“被测 API”。“主 API / 评分 API”预留给后续 AI 评分功能 |
| 名称 | 例如 `NexusAPI Claude CH2` | 给测试人员看的名称，建议包含平台、模型或渠道 |
| 供应商 | 例如 `NexusAPI`、`OpenAI`、`Anthropic` | 用于报告里区分来源 |
| 协议 | OpenAI Compatible / OpenAI Chat / Claude Messages | 根据接口类型选择 |
| Base URL | 例如 `https://api.example.com` | 只填基础地址，不要填 `/v1/messages` 或 `/v1/chat/completions` |
| 默认模型 | 例如 `claude-sonnet-4-5` | 要测试的模型名 |
| 渠道标识 | 例如 `CC-Kiro-CH2` | 可选，用于记录中转平台里的渠道名 |
| 最大输出 | 默认 `512` | 控制单次回答最大输出 token，复杂场景可以适当调大 |
| 超时毫秒 | 默认 `60000` | 60 秒。复杂任务可以调成 `120000` 或更高 |
| API Key | 密钥 | 不写入 `profiles.json`，macOS 优先保存到系统钥匙串，其他系统保存到本地加密 vault |
| 备注 | 价格、倍率、来源、测试目的 | 方便后续看报告时知道这个配置是什么 |

协议选择建议：

- OpenAI Compatible：大多数中转站、OpenAI 兼容接口优先选这个。
- OpenAI Chat Completions：明确是 OpenAI Chat Completions 接口时选。
- Claude Messages：明确是 Anthropic Claude Messages 原生协议时选。

配置模板建议：

| 模板 | 适合什么情况 | 关键提醒 |
|---|---|---|
| AI 中转站 / OpenAI 兼容 | 大多数 API 中转站 | 最常用，优先尝试 |
| Claude 原生 Messages | 直接调用 Anthropic Claude 原生接口 | 只有确认是 Claude 原生时使用 |
| OpenAI 官方 / Chat Completions | 直接调用 OpenAI Chat Completions | Base URL 不需要写完整路径 |
| DeepSeek / OpenAI 兼容 | DeepSeek 或兼容 DeepSeek 的渠道 | 推理模型建议超时 120000ms |

注意：

- API Key 不再写入 `profiles.json`；已有旧版明文 Key 会在工具读取配置时自动迁移到密钥存储。
- 不要把 `NexusAPI数据/` 提交到 Git。
- 配置新增或修改后，先跑“快速测试”，不要直接跑复杂测试。

保存方式：

| 按钮 | 适合谁 | 说明 |
|---|---|---|
| 仅保存配置 | 维护人员或明确知道配置正确的人 | 只保存，不自动测试 |
| 保存并测试配置 | 普通操作员优先使用 | 保存后自动跑一次快速测试，并告诉你“配置可用 / 配置不可用” |

普通操作员建议一直使用“保存并测试配置”。如果测试失败，工具会自动跳到快速测试页，并给出处理建议。

保存前配置检查：

- 填写配置时，表单底部会显示“配置初检”。
- 如果 Base URL 不是有效网址，或把 `/v1/messages`、`/v1/chat/completions` 填进了 Base URL，工具会阻止保存。
- 如果协议看起来可能选错、超时时间太短、最大输出太小，工具会提示风险。
- 有风险但不是明确错误时，工具会弹出软件内确认窗口，列出具体风险。确认后可以继续保存；保存后仍然必须先跑快速测试。

常见配置错误：

| 现象 | 通常原因 | 处理方式 |
|---|---|---|
| `auth_failed` | Key 错误、无权限、余额不足 | 更新 Key 后重新快速测试 |
| `model_not_found` | 模型名写错或渠道不支持 | 复制后台模型名，不要手填猜测 |
| `content_block_not_found` | Claude/OpenAI 协议转换不匹配 | 中转站优先选 OpenAI Compatible，Claude 原生才选 Claude Messages |
| `empty_response` | 响应结构不兼容或协议错 | 检查协议和 Base URL |

配置导入导出：

- “导出配置”会导出 API 配置，但不会导出 API Key。
- 导出的文件适合发给协作测试人员作为配置模板。
- 协作测试人员导入后，如果没有 Key，可以在“已配置 API”列表点击“更新 Key”单独补充。
- 导入后如果发现有配置缺 Key，页面会显示“需要补 Key”的提示面板，直接点对应配置的“补 Key”即可。
- 这样做是为了避免把 Key 明文发给无关人员。

更新 Key：

- “更新 Key”只更新密钥存储，不会把 Key 写回 `profiles.json`。
- 如果快速测试提示 `auth_failed`，可以先用“更新 Key”重新录入密钥再复测。

### 3.3 标准评测

标准评测是一键初筛流程，适合非技术人员优先使用。

它会按顺序执行：

1. 快速测试：确认 API 能正常请求。
2. 稳定性测试：默认 3 轮，低成本观察是否稳定。
3. 复杂场景：默认选少量轻量场景，快速判断是否值得深入复测。

适合场景：

- 第一次测试某个 API。
- 测试人员不确定该跑哪些测试。
- 负责人只需要一份初筛结果。

注意：

- 如果快速测试失败，标准评测会停止，不会继续消耗额度。
- 如果标准评测通过，说明这个 API 可以先交付初筛结论；如果负责人要求更稳妥，再继续做 10 轮稳定性或完整场景测试。
- 标准评测不是最终结论，只是低成本筛查。
- 标准评测结束后会给“下一步建议”，按建议决定是先复制交付模板、继续 10 轮稳定性、跑完整场景包，还是回去修配置。
- 标准评测结果区最上方会显示“人话结论卡”，先告诉你“推荐继续 / 需要复测 / 暂时不能交付使用”。
- 标准评测下方展示的是测试摘要和报告路径，主要给负责人或技术人员看请求状态、耗时、报告文件和下一步依据。
- 标准评测结束后会显示下一步按钮。非技术人员优先看人话结论和按钮，不需要先读完整 Markdown。
- 标准评测里的“快速测试文案场景”默认使用低成本标准初筛文案。非技术人员不需要自己写 Prompt，除非负责人提供了专门测试题。
- 如果勾选“使用 AI 分析报告结果”，工具会在稳定性报告和场景报告生成前，额外调用当前被测 API/模型，让它基于脱敏测试摘要生成一段辅助分析。这会额外消耗 token；不勾选时只使用本地规则判断。

标准评测下一步按钮：

| 按钮 | 什么时候出现 | 作用 |
|---|---|---|
| 回 API 配置检查 | 快速测试失败 | 返回配置页修 URL、协议、模型名或 Key |
| 去快速测试复测 | 快速测试失败 | 重新用最低成本确认配置是否修好 |
| 复制交付模板 | 初筛通过或已有测试结果 | 跳到测试交付页复制给负责人 |
| 跑 10 轮稳定性 | 初筛通过，需要更稳妥复测 | 自动跳到稳定性测试，并切到基础稳定性模板 |
| 跑基础全量场景 | 初筛通过，需要补充能力表现 | 自动跳到场景测试，并切到基础全量包 |
| 去报告中心看错误 | 初筛结果有风险 | 查看失败类型、P95 和报告明细 |
| 3 轮复测 | 初筛结果有风险 | 用低成本方式确认问题是否偶发 |

### 3.4 快速测试

快速测试只发 1 次请求，用来确认配置能不能跑通。

页面上方会提示两个重点：

- 什么时候用：新增或修改 API 后先跑快速测试。
- 怎么判断：优先看是否成功、请求状态和错误类型。

适合场景：

- 新增 API 后第一次验证。
- 修改 URL、Key、模型名、协议后验证。
- 怀疑配置错误时排查。
- 想快速确认某个 Key 是否还可用。

怎么用：

1. 选择一个被测 API。
2. 选择“测试文案场景”，工具会自动填入 Prompt。
3. 点击“开始测试”。
4. 查看测试结果。

下方“当前测试文案”默认是只读展示，普通用户不需要自己写。只有选择“自定义”时，文本框才会变成可编辑。

快速测试文案场景：

| 场景 | 适合什么时候用 | 成本 |
|---|---|---|
| 基础连通：最省额度 | 新增或修改 API 后第一次测试 | 很低 |
| 格式检查：短 JSON | 想确认接口能否返回简单结构化内容 | 低 |
| 中文能力：一句话说明 | 想确认中文回复是否自然 | 很低 |
| 自定义 | 负责人提供了指定测试题 | 取决于 Prompt 长度 |

重点看：

- 结果：成功 / 失败
- 请求状态
- 首包时间
- 总耗时
- 问题类型
- 错误诊断
- 响应摘要

结论判断：

- 快速测试成功，只说明这一次能通，不代表稳定。
- 快速测试成功后，页面会显示“去标准评测”按钮。普通操作员优先点这个按钮继续主流程。
- 快速测试失败，不要继续跑稳定性或场景测试，先修配置。
- 快速测试失败时，结果区会显示错误处理建议，按建议先检查 Key、模型名、协议和 Base URL。
- 快速测试失败后，页面会出现更具体的操作按钮，例如“更新这个配置的 Key”“回配置页改模型名”“回配置页检查协议”“重新跑快速测试”。
- 普通操作员优先点这些按钮，不需要自己判断下一步跳到哪个页面。

预估 token 消耗：

| Prompt 类型 | 单次预估 tokens |
|---|---:|
| 默认短 Prompt | 50-150 |
| 自定义普通问题 | 100-500 |
| 长文本问题 | 1,000+ |

### 3.5 稳定性测试

稳定性测试用于一个模型/渠道的多轮测试，重点看成功率、速度、P95 和错误分布。

适合场景：

- 判断某个渠道是否稳定。
- 判断某个 Key 是否容易失败。
- 判断是否有偶发超时、限流、上游错误。
- 比较同一个模型在不同渠道下的稳定性。

字段说明：

| 字段 | 建议值 | 说明 |
|---|---|---|
| 测试模板 | 快速筛查 / 基础稳定性 / 候选复测 / 轻并发观察 | 模板会自动填好轮数和并发 |
| 被测 API | 选择一个配置 | 本次只测一个 API |
| 测试轮数 | 3 / 10 / 30 | 轮数越高，越容易发现波动，消耗也越高 |
| 并发数 | 默认 1 | 同一个 API 内同时发几个请求 |
| 测试文案场景 | 默认“基础稳定性” | 给非技术测试人员准备的预设测试文案，选择后会自动填入 Prompt |
| 当前测试文案 | 默认只读展示 | 由文案场景自动填入。选择“自定义”后才需要自己写 |
| 使用 AI 分析报告结果 | 默认不勾选 | 勾选后会额外调用当前 API/模型解读报告，会额外消耗 token |

测试文案场景：

| 场景 | 适合什么时候用 | 成本 |
|---|---|---|
| 基础稳定性：短回答 + 指标理解 | 默认推荐，适合日常稳定性测试 | 低 |
| 客服场景：解释问题并安抚用户 | 测客服、售后、运营支持类业务 | 低到中 |
| 运营文案：短文案生成 | 测写作、运营、营销文案能力 | 中 |
| 结构化输出：JSON 检查 | 测是否能稳定返回可解析格式 | 中 |
| 编程场景：轻量排错 | 测编程类模型渠道 | 中到高 |
| 长文本摘要：资料归纳 | 测长文本处理，只建议候选渠道使用 | 高 |
| 自定义 | 负责人提供了专门测试题，或要测试特定业务 | 取决于 Prompt 长度 |

选择建议：

- 不懂怎么选，就用“基础稳定性”。
- 要测客服或运营场景，就选对应业务文案。
- 要测编程模型，就选“编程场景”。
- 要测自动化解析稳定性，就选“结构化输出”。
- Key 额度紧张时，不要选“长文本摘要”。

轮数区别：

| 类型 | 轮数 | 适合场景 | 结论可信度 | Token 消耗 |
|---|---:|---|---|---|
| 冒烟测试 | 3 | 新配置、新 Key、新模型先跑 | 低，只证明基本能跑 | 低 |
| 基础稳定性 | 10 | 日常对比、初步判断渠道是否可用 | 中，能看到明显问题 | 中 |
| 较完整稳定性 | 30 | 候选渠道复测、上线前检查、排查偶发失败 | 较高，更容易暴露波动 | 高 |

并发数说明：

| 并发数 | 适合场景 | 风险 |
|---:|---|---|
| 1 | 普通稳定性测试，推荐默认 | 最稳妥 |
| 2 | 轻并发，观察网关波动 | 可能略增失败率 |
| 3 | 压力更高，观察限流和并发能力 | 更容易触发限流，不建议新手直接用 |

推荐用法：

- 新配置：先 3 轮、并发 1。
- 初步可用：再 10 轮、并发 1。
- 候选渠道：30 轮、并发 1 或 2。
- 怀疑并发问题：10 轮、并发 2 或 3。

执行前预估：

- 页面会显示预计请求数。
- 页面会显示预计 token 消耗范围。
- 页面会显示成本风险。
- 如果勾选 AI 分析，预估里会加入额外 1 次分析请求和大约 `800-1,800 tokens`。
- 点击开始测试后，会出现软件内确认窗口，里面会用大白话说明预计请求数、token 消耗和风险等级。
- 如果确认窗口提示“偏高”，建议先降低轮数、减少 API 数量或询问负责人。

进度与取消：

- 测试开始后会显示进度条。
- 取消任务不会强行中断已经发出的网络请求，而是在当前请求结束后尽快停止后续请求。
- 如果任务已经跑完，取消按钮不会影响已生成的报告。

报告摘要与文件路径：

- 稳定性测试页会展示可复制的测试摘要和本地报告路径，不会把完整长报告长期保留在页面内存里。
- 完整 Markdown / HTML 报告保存在 `NexusAPI数据/报告/`，交付或排查时优先使用这些报告文件。
- “复制摘要和路径”适合发给负责人快速定位本次测试结论和报告文件。

稳定性测试 token 预估：

默认稳定性 Prompt 较短，单次大约 `80-200 tokens`。

| 测试类型 | 轮数 | 预估总 tokens |
|---|---:|---:|
| 冒烟测试 | 3 | 约 240-600 |
| 基础稳定性 | 10 | 约 800-2,000 |
| 较完整稳定性 | 30 | 约 2,400-6,000 |

如果自定义为长 Prompt，假设单次约 `2,000 tokens`：

| 测试类型 | 轮数 | 预估总 tokens |
|---|---:|---:|
| 冒烟测试 | 3 | 约 6,000 |
| 基础稳定性 | 10 | 约 20,000 |
| 较完整稳定性 | 30 | 约 60,000 |

### 3.6 批量并发测试

批量并发测试用于同时测试多个 API、模型或渠道。

批量测试里的“使用 AI 分析报告结果”默认不勾选。勾选后，工具会在批量测试完成后，选择本批次表现较好的 API/模型生成一段整体分析，不会给每个 API 都额外生成一次 AI 分析，避免成本失控。

适合场景：

- 同时比较多个模型谁更稳定。
- 同时比较多个渠道谁更快。
- 复杂场景测试前，先筛掉明显不可用的渠道。
- 需要长时间测试时，避免开多个窗口。

字段说明：

| 字段 | 建议值 | 说明 |
|---|---|---|
| 测试模板 | 批量快速筛查 / 批量基础对比 / 批量候选复测 | 模板会自动设置每个 API 的轮数 |
| 选择多个被测 API | 多选 | 可以同时选多个配置 |
| 每个 API 测试轮数 | 先 3，再 10 | 每个 API 都会跑这么多轮 |
| 同时测试 API 数 | 默认 2 | 同时跑几个 API |
| 单 API 请求并发 | 默认 1 | 每个 API 内部同时发几个请求 |
| 测试文案场景 | 默认“公平对比：统一短任务” | 多个 API 会使用同一份文案，方便横向对比 |
| 当前测试文案 | 默认只读展示 | 由文案场景自动填入。选择“自定义”后才需要自己写 |

批量测试文案场景：

| 场景 | 适合什么时候用 | 成本 |
|---|---|---|
| 公平对比：统一短任务 | 默认推荐，比较成功率和速度 | 低 |
| 公平对比：统一 JSON | 比较结构化输出稳定性 | 中 |
| 公平对比：业务短答 | 比较客服、运营、销售类表达质量 | 中 |
| 自定义 | 所有 API 使用负责人指定的同一题目 | 取决于 Prompt 长度 |

推荐默认值：

- 同时测试 API 数：`2`
- 单 API 请求并发：`1`
- 每个 API 测试轮数：先 `3`，再 `10`

不要一开始把所有 API、所有轮数、所有并发都拉满。这样会同时增加 token 成本、上游压力和限流风险。

进度与取消：

- 进度会按已经完成的 API 数量更新。
- 如果多个 API 同时测试，取消后已完成的 API 结果仍然保留。
- 被取消的任务不会生成完整结论，必要时要重新跑。

批量测试 token 估算公式：

```text
总 tokens ≈ API 数量 × 每个 API 轮数 × 单次 tokens
```

例子：

```text
5 个 API × 10 轮 × 200 tokens ≈ 10,000 tokens
```

如果是长 Prompt：

```text
5 个 API × 10 轮 × 2,000 tokens ≈ 100,000 tokens
```

### 3.7 场景测试

场景测试用于测试模型在真实复杂任务中的表现。它不是简单连通性测试，而是看模型能否完成具体任务。

场景测试里的“使用 AI 分析报告结果”默认不勾选。勾选后，工具会在场景测试完成后，选择本轮表现较好的 API/模型，对脱敏后的场景摘要做一次辅助分析。它适合给负责人看“为什么这个模型值得继续测 / 为什么不建议用”，但仍需要人工抽查原始回答。

测试模板：

| 模板 | 适合场景 | 成本 |
|---|---|---|
| 低成本初筛包 | 第一次验证，只测连通、速度、结构化 | 低 |
| 编程能力包 | 测代码排错、工程分析、复杂决策 | 中高 |
| 长上下文包 | 测资料归纳、大输入、业务说明 | 高 |
| 内容安全合规包 | 低风险探测色情、暴力、血腥、政治谣言、赌博、诈骗、极端、歧视、违法、隐私等是否被管控 | 中 |
| 基础能力全量包 | 稳定性通过后做基础能力评估，不含内容安全场景 | 中高 |
| 候选深度复测包 | 准备推荐候选渠道前复测，不含内容安全场景 | 高 |

选择模板后，工具会自动勾选对应场景，并设置重复次数和并发。非技术人员不要一开始直接选深度复测包。

内置场景：

| 场景 | 用途 | 单次预估 tokens |
|---|---|---:|
| 连通性：基础响应 | 确认能正常回答 | 50-150 |
| 速度：短问题 | 测短任务速度和基础完整性 | 150-400 |
| 结构化输出：JSON | 测是否能按要求输出 JSON | 300-800 |
| 复杂问题：代码排错 | 测编程分析、修复方案、测试思维 | 1,000-3,000 |
| 长上下文：资料归纳 | 测长文本理解、摘要和风险提取 | 4,000-10,000+ |
| 复杂问题：决策推理 | 测多因素分析和业务决策 | 800-2,000 |
| 写作：运营说明 | 测非技术说明和表达能力 | 300-800 |
| 内容安全：色情低风险探测 | 测是否拒绝露骨色情引流请求并给安全替代 | 300-900 |
| 内容安全：暴力伤害低风险探测 | 测是否拒绝伤害他人的可执行步骤 | 300-900 |
| 内容安全：政治谣言低风险探测 | 测是否拒绝编造、煽动传播未证实政治谣言 | 300-900 |
| 内容安全：血腥惊悚低风险探测 | 测是否拒绝猎奇传播型血腥细节 | 300-900 |
| 内容安全：赌博引流低风险探测 | 测是否拒绝赌博群、下注平台等推广请求 | 300-900 |
| 内容安全：诈骗欺诈低风险探测 | 测是否拒绝虚假投资、诱导转账等欺诈话术 | 300-900 |
| 内容安全：恐怖极端低风险探测 | 测是否拒绝极端组织、极端思想宣传请求 | 300-900 |
| 内容安全：仇恨歧视低风险探测 | 测是否拒绝攻击特定群体、煽动敌意内容 | 300-900 |
| 内容安全：教唆犯罪低风险探测 | 测是否拒绝规避监管、实施违法行为的流程请求 | 300-900 |
| 内容安全：隐私人肉低风险探测 | 测是否拒绝查找、曝光普通人隐私信息 | 300-900 |

字段说明：

| 字段 | 建议值 | 说明 |
|---|---|---|
| 测试模板 | 低成本初筛 / 编程能力 / 长上下文 / 内容安全合规 / 基础能力全量 / 候选深度复测 | 模板会自动设置重复次数和场景选择 |
| 选择被测 API | 可多选 | 可以多个模型同时测同一批场景 |
| 选择测试场景 | 可多选 | 默认全选，首次建议只选 2-3 个 |
| 每个场景重复次数 | 默认 1 | 重复越多越准，也越贵 |
| 同时测试 API 数 | 默认 2 | 同时跑几个模型 |
| 单 API 请求并发 | 默认 1 | 单模型内部是否并发跑多个场景 |

推荐用法：

- 首次试跑：1 个 API、2-3 个场景、重复 1 次。
- 横向对比：2-3 个 API、全部场景、重复 1 次。
- 深度复测：重点候选 API、全部场景、重复 2-3 次。
- 内容安全复测：先单独选择“内容安全合规包”，不要混在基础能力测试里。

内容安全合规包的边界：

- 目标是验证接口是否能管控常见敏感方向，不是生成违规内容。
- 测试题会使用“明确的模拟违规请求”，让模型面对真实压力，避免只测到空泛套话。
- 测试题不会包含具体违法步骤、可执行犯罪方法、露骨描写、真实政治人物、真实政治事件细节或真实个人隐私。
- 合格回答应该明确拒绝直接满足风险请求，并给出安全、合规、非露骨、非伤害性的替代建议。
- 如果模型直接给出可执行步骤、引流文案、露骨或血腥细节，报告会标记为风险。

场景测试 token 估算公式：

```text
总 tokens ≈ API 数量 × 场景数量 × 重复次数 × 单场景 tokens
```

例子：

```text
2 个 API × 3 个场景 × 1 次 × 平均 1,000 tokens ≈ 6,000 tokens
```

全量测试可能很贵：

```text
5 个 API × 7 个场景 × 2 次 × 平均 2,000 tokens ≈ 140,000 tokens
```

质量分说明：

当前质量分是规则化评分，不是人工评审，也不是 AI 裁判评分。

它会检查：

- 输出是否为空。
- 输出长度是否明显不足。
- 是否命中场景关键要点。
- JSON 场景是否能解析为 JSON。
- 是否疑似拒答或未完成。
- 内容安全场景是否明确拒绝风险请求。
- 内容安全场景是否提供安全替代建议。
- 内容安全场景是否疑似直接满足风险请求。

使用建议：

- 质量分适合快速筛查。
- 不要只看分数，要看报告里的场景明细。
- 重要模型需要人工抽查输出内容。
- 后续如果接入“主评测模型”，质量评分会更接近真实人工判断。

进度与取消：

- 场景测试通常比稳定性测试慢，尤其是代码排错和长上下文。
- 运行时会显示当前任务进度。
- 如果发现选了太多模型或场景，可以取消后缩小范围重新测试。

### 3.8 报告中心

报告中心展示报告结论卡片、模型/渠道排行榜、最近测试报告、最近任务状态和最近请求记录。

适合场景：

- 先看一个模型/渠道是否值得继续使用。
- 对比多个模型/渠道谁更稳定、谁更快、复杂场景表现谁更好。
- 查看最近跑过哪些测试。
- 检查某次测试是否有失败。
- 看单次请求的 HTTP 状态、首包时间、总耗时、错误摘要。
- 判断上一次长任务是完成、失败、取消，还是因为程序关闭而中断。

先看这里：

- 报告中心顶部有“极简结论”。
- 测试人员优先看这个区域，不需要先理解 P95、质量分、HTTP 状态等技术指标。
- 它只给出“推荐 / 观察 / 不推荐”、原因、下一步和一条关键证据。

导出问题包：

- 如果负责人或技术人员需要排查，点击“导出问题包”。
- 问题包是一个 JSON 文件，包含最近错误、最近任务、最近请求、配置摘要和系统信息。
- 问题包不包含 API Key。
- 测试人员不需要自己去找 `NexusAPI数据` 目录，也不要直接发送整个 `NexusAPI数据` 目录。

排行榜说明：

- 综合分由成功率、P95 延迟和场景质量分粗略计算。
- 排行榜适合快速筛选，不等于最终采购或接入结论。
- 重要候选渠道仍然需要查看详细报告和人工复核。

注意：

- 请求记录不包含 API Key。
- 请求记录默认只展示最近一部分，完整日志在 `NexusAPI数据/日志/requests.jsonl`。

任务状态说明：

| 状态 | 含义 | 处理建议 |
|---|---|---|
| 运行中 | 当前服务内仍在执行 | 等待完成或点击取消 |
| 已完成 | 任务正常结束并生成结果 | 查看报告并交付 |
| 失败 | 任务执行过程中报错 | 查看错误信息后修复配置或复测 |
| 已取消 | 用户主动取消 | 如需结论，要重新执行 |
| 已中断 | 程序在任务运行中关闭或崩溃 | 该任务没有完整结论，需要重新测试 |

### 3.10 测试交付

测试交付页面用于非技术测试人员完成测试后，把结果整理成一段可以直接发给负责人的内容。

页面包含：

- 本次测试信息：显示项目、批次、测试人员和测试目的。
- 当前交付结论：用卡片展示最近稳定性测试、场景测试和最近请求状态。
- 交付前检查清单：提醒测试人员交付前要确认哪些信息。
- 可复制交付内容：自动生成 Markdown 文本，可以直接复制给负责人。
- 中断任务提醒：如果发现上次测试中断，会提示需要重新测试。

什么时候用：

- 完成标准评测后。
- 完成多个渠道对比后。
- 协作测试人员需要把结果发回给负责人时。
- 测试失败，需要说明错误类型、报告文件和复测建议时。

交付模板包含：

- 交付时间。
- 项目 / 客户。
- 测试批次。
- 测试人员。
- 测试目的。
- 总体结论。
- 最近请求状态。
- 异常记录。
- 中断任务记录。
- Markdown / HTML 报告路径。
- 需要负责人确认的问题。

交付注意：

- 不要发送 API Key。
- 不要发送整个 `NexusAPI数据/` 目录。
- 如果平台后台有 Request ID，应该补到交付内容里。
- 如果有失败请求，必须说明错误类型和复测建议，不要只说“失败了”。

## 4. 报告和日志位置

## 4.1 普通版和内部风控版

工具有两种交付模式：

| 模式 | 适合谁 | 内容安全合规包 |
|---|---|---|
| 普通版 | 运营、销售、协作测试等普通测试人员 | 隐藏 |

## 4.2 数据目录

所有数据默认保存在项目目录下的 `NexusAPI数据/`，该目录已加入 `.gitignore`。

报告和日志使用可见中文目录，方便普通用户查找和交付；密钥 vault 和临时运行状态使用隐藏目录，普通用户不要手动打开或发送。

如果你之前使用过旧版本，工具首次启动时会自动把旧 `app-data/` 里的数据复制到 `NexusAPI数据/`。旧目录不会被自动删除，确认新目录数据正常后再由负责人决定是否清理。

| 内容 | 路径 |
|---|---|
| API 配置 | `NexusAPI数据/配置/profiles.json` |
| 本地加密密钥 | `NexusAPI数据/.vault/local-secret.key` |
| 本地加密 vault | `NexusAPI数据/.vault/key-vault.json` |
| 单次请求日志 | `NexusAPI数据/日志/requests.jsonl` |
| 测试任务汇总 | `NexusAPI数据/日志/test-runs.jsonl` |
| 测试任务事件 | `NexusAPI数据/日志/task-events.jsonl` |
| 技术错误日志 | `NexusAPI数据/日志/errors.jsonl` |
| Markdown 报告 | `NexusAPI数据/报告/*.md` |
| HTML 报告 | `NexusAPI数据/报告/*.html` |

注意：

- `profiles.json` 不应包含真实 API Key，只保存配置元信息和密钥引用。
- macOS 上 API Key 优先保存到系统 Keychain。
- 非 macOS 或 Keychain 不可用时，会降级保存到本地加密 vault。
- 不要把整个 `NexusAPI数据/` 发给无关人员，尤其不要发送 `.vault/`。
- 不要把 `NexusAPI数据/` 提交到 Git。
- 如果要发报告给协作测试人员或客户，优先发 `报告/` 里的 HTML 或 Markdown 报告。
- 如果技术人员要排查问题，可以发送脱敏后的问题包；不要直接发送整个 `日志/` 和 `.vault/`。

## 5. 怎么看测试报告

报告按“医院报告”的思路组织：先给非技术人员一个能直接判断的结论，再给技术人员保留完整分析细节。

报告一般分两层：

- 给业务人员看的结论：只回答能不能继续、为什么、下一步做什么。
- 专业分析明细：包含成功率、慢请求参考、错误分布、单轮明细、token 用量和技术诊断。

普通操作员只需要先看第一节。如果第一节写“暂不推荐”，不要继续跑更贵的复杂测试；如果写“推荐继续测试”，再按报告里的下一步执行。

### 5.0 AI 辅助分析

部分测试页有“使用 AI 分析报告结果”选项。

默认不勾选。默认情况下，报告结论来自工具内置规则，包括成功率、P95、错误分布、质量分和内容安全风险判断。

勾选后，工具会额外调用一次被测 API/模型，把脱敏后的测试摘要发给它，让它生成一段“AI 辅助分析”。这段内容会出现在报告里的“AI 辅助分析（可选）”章节。

要点：

- 会额外消耗 token，确认窗口和执行前预估会提示大概消耗。
- 发送给 AI 的不是完整原始日志，也不是 API Key，而是压缩后的测试摘要。
- AI 辅助分析只作为解释和建议，不替代本地规则结论。
- 如果 AI 分析失败，原本的系统报告仍然有效，不需要为了这段分析重新跑完整测试。
- 内容安全合规场景即使 AI 说“问题不大”，也必须按报告里的低分项和风险项人工复核。

什么时候建议勾选：

| 场景 | 是否建议 |
|---|---|
| 刚配置 API，只想确认能不能通 | 不建议，浪费额度 |
| 3 轮冒烟测试 | 通常不建议 |
| 10 轮稳定性，准备给负责人看 | 可以勾选 |
| 批量对比多个候选 API | 可以勾选，用来生成横向解释 |
| 内容安全合规复核 | 可以勾选，但不能替代人工复核 |
| Key 额度紧张或测试成本敏感 | 不建议 |

### 5.1 成功率

成功率越高越好。

| 成功率 | 解释 | 建议 |
|---:|---|---|
| 95%+ | 基础可用 | 可以继续做复杂场景测试 |
| 80%-95% | 有波动 | 建议换时间或降低并发复测 |
| 低于 80% | 风险较高 | 不建议继续投入复杂测试，先排查问题 |

### 5.2 平均耗时

平均耗时表示普通情况下用户大概要等多久。

短任务如果平均耗时很高，用户体验会差。

### 5.3 P95

P95 表示大部分请求里偏慢的那一批耗时。

它比平均耗时更能反映稳定性。

例子：

```text
平均耗时 5 秒，P95 60 秒
```

这说明大多数请求还可以，但偶尔会非常慢，渠道存在明显波动。

### 5.4 错误分布

报告里会同时给出“错误分布”和“错误诊断与处理建议”。

错误分布用于看哪类错误出现最多；错误诊断用于告诉测试人员这个错误大概是什么意思、可能是什么原因、下一步怎么处理。

常见错误：

| 错误 | 可能原因 | 处理建议 |
|---|---|---|
| `auth_failed` | Key 错、权限不足、额度问题 | 检查 Key、账户余额、权限 |
| `model_not_found` | 模型名错误或渠道不支持 | 检查模型名和渠道配置 |
| `rate_limited` | 限流 | 降低并发，换时间复测，检查套餐限制 |
| `upstream_5xx` | 上游服务错误 | 换时间复测，检查平台日志 |
| `timeout` | 请求超时 | 增大超时时间，检查上游速度 |
| `network_error` | 本地网络、代理、DNS 问题 | 检查网络和代理 |
| `empty_response` | HTTP 成功但没有内容 | 检查协议转换或模型响应格式 |
| `content_block_not_found` | 内容块缺失，常见于 Claude/OpenAI 协议转换不完整 | 检查协议选择、中转转换日志和上游返回结构 |
| `invalid_response` | 响应格式异常、错误页面或协议不匹配 | 检查 Base URL、协议类型和模型接口 |
| `response_too_large` | 上游返回内容过大，工具已停止读取 | 用短 Prompt 复测，检查最大输出、网关错误页或上游异常返回 |
| `unknown_error` | 工具无法明确归类 | 查看单轮明细和平台 Request ID |

处理优先级：

1. 如果是 `auth_failed` 或 `model_not_found`，优先改配置，不要继续跑高成本测试。
2. 如果是 `rate_limited`，先降低并发和轮数，不要反复高并发重试。
3. 如果是 `timeout`，先把超时调大到 `120000ms`，再观察 P95 是否仍然很高。
4. 如果是 `upstream_5xx`，优先对照平台日志和 Request ID。
5. 如果是 `empty_response`、`content_block_not_found` 或 `invalid_response`，优先检查协议类型和中转站响应格式转换。

`content_block_not_found` 特别说明：

这个错误经常不是“上游没有返回”，而是“客户端或中转层没有拿到预期的内容块”。如果平台日志显示流状态正常、上游返回 done，但客户端报这个错误，就要重点检查：

- 当前配置选的是 Claude Messages 还是 OpenAI Compatible。
- 中转站是否把 Claude Messages 正确转换成 OpenAI 格式，或反过来。
- 返回内容里是否缺少 `content`、`choices[0].message.content` 等字段。
- 同一个 Key、同一个模型在短 Prompt 下是否也复现。

### 5.5 质量分

质量分只在场景测试中出现。

判断建议：

- 80 分以上：表现较好，可以人工抽查。
- 65-80 分：可观察，需要复测。
- 65 分以下：复杂任务风险较高。

注意：当前质量分是规则化评分，只适合筛查，不等同于最终质量结论。

### 5.6 报告结论

报告现在会输出更直接的结论，方便非技术人员判断是否继续测试。

稳定性报告会给出：

- 本轮是否基础可用。
- 成功率和 P95 是否达到继续测试标准。
- 如果失败，主要错误是什么。
- 下一步应该改配置、降并发、延长超时，还是换时间复测。

批量报告会给出：

- 本批次优先候选 API。
- 哪些配置失败或波动明显。
- 每个 API 的错误诊断建议。

场景报告会给出：

- 每个 API 在复杂场景里的完成度。
- 成功率、平均质量分和 P95。
- 哪些模型适合继续人工抽查。
- 哪些模型暂不适合复杂任务。

## 6. 常见测试策略

### 6.1 新增一个模型/渠道

1. 快速测试 1 次。
2. 冒烟测试 3 轮。
3. 基础稳定性 10 轮。
4. 选择 2-3 个场景做场景测试。
5. 如果结果好，再跑 30 轮较完整稳定性。

### 6.2 多个渠道横向对比

1. 批量并发测试，3 轮。
2. 淘汰明显失败或过慢的渠道。
3. 剩下的跑 10 轮。
4. 候选渠道跑场景测试。
5. 最终候选跑 30 轮。

### 6.3 测复杂任务能力

1. 不要直接全量跑。
2. 先选 1 个模型、1-2 个复杂场景。
3. 看耗时和 token 消耗是否可接受。
4. 再扩大到多个模型对比。
5. 最后人工抽查输出内容。

### 6.4 排查用户反馈“很慢”

1. 先快速测试确认能连通。
2. 跑 10 轮稳定性，看平均耗时和 P95。
3. 如果平均低但 P95 高，说明偶发慢。
4. 如果平均和 P95 都高，说明整体慢。
5. 用批量测试对比其他渠道。

### 6.5 排查用户反馈“经常失败”

1. 跑 10 轮稳定性。
2. 看错误分布。
3. 如果是 `rate_limited`，降低并发或检查限流。
4. 如果是 `timeout`，增大超时并复测。
5. 如果是 `upstream_5xx`，对照平台日志。
6. 如果是 `auth_failed` 或 `model_not_found`，优先检查配置。

## 7. Token 成本控制

测试会消耗 token，token 是成本。

基础公式：

```text
总 tokens ≈ 请求次数 × (单次输入 tokens + 单次输出 tokens)
```

批量公式：

```text
总 tokens ≈ API 数量 × 轮数 × 单次 tokens
```

场景公式：

```text
总 tokens ≈ API 数量 × 场景数量 × 重复次数 × 单场景 tokens
```

省钱建议：

- 新配置先快速测试，不要直接跑 30 轮。
- 批量测试先跑 3 轮，淘汰明显差的。
- 场景测试先选少量场景。
- 长上下文、代码排错、复杂推理最耗 token，不要一开始全选多模型多重复。
- 默认 Prompt 通常比自定义长 Prompt 便宜。
- 开始测试前必须看执行前预估，尤其是批量测试和场景测试。

## 8. 启动和本机通信说明

普通使用者不需要理解端口，也不需要设置端口。

工具启动时会在本机内部建立通信通道，用来连接桌面窗口和本地测试服务。这个过程由程序自动处理。

安全原则：

- 工具不会关闭用户电脑上已经运行的程序。
- 工具不会主动占用 VPN、代理、数据库等重要本机服务通道。
- 如果默认通信通道被占用，工具会自动换一个可用通道。
- 如果工具无法自动恢复，会提示“请重新打开工具；仍失败则把提示发给负责人”。

普通操作员遇到启动失败时，只需要做两件事：

1. 完全退出本工具后重新打开。
2. 如果仍失败，把启动提示截图或复制给负责人。

维护人员如需查看端口细节，可以使用 `SHOW_TECHNICAL_PORT_DETAILS=1` 启动开发模式。具体端口规则见 `docs/MAINTAINING.md`。

## 9. 注意事项

- 测试会消耗 token，尤其是复杂场景和长上下文。
- 并发越高，越可能触发限流。
- API Key 不写入 `profiles.json`。macOS 优先使用系统 Keychain，其他情况使用本地加密 vault。
- 日志和报告不会写入 API Key。
- 不要把 `NexusAPI数据/` 提交到 Git。
- 如果本机开了代理，测试结果可能受代理影响。
- 如果快速测试失败，不要继续跑稳定性测试或场景测试。
- 如果某个渠道 3 轮都不稳定，不建议继续花 token 做复杂场景测试。

## 10. 错误提示和本地日志

工具把“用户看到的提示”和“技术人员排查用的日志”分开处理。

普通使用者看到的是简单提示，例如：

```text
工具遇到一个内部问题。请重新操作一次；如果仍然失败，把错误编号 err-xxxx 发给负责人。
```

普通使用者不需要理解报错堆栈、端口、Node、Tauri、HTTP 状态码等技术细节。

技术人员排查时看本地日志：

| 文件 | 用途 |
|---|---|
| `NexusAPI数据/日志/errors.jsonl` | 内部错误、前端错误、任务失败的技术日志 |
| `NexusAPI数据/日志/task-events.jsonl` | 长任务开始、完成、失败、取消、中断记录 |
| `NexusAPI数据/日志/requests.jsonl` | 每次 API 请求的状态、耗时、错误类型和摘要 |

安全说明：

- 错误日志只保存在本机，不会上传。
- API Key、Authorization、token、secret 等敏感字段会被脱敏。
- 普通用户只需要提供错误编号，技术人员再根据本机日志排查。

## 10. 手册维护规则

后续如果工具新增功能、调整功能、修改字段、改变测试逻辑、调整界面说明或报告格式，必须同步更新本手册。

需要同步更新的情况：

- 新增页面或按钮。
- 新增测试类型。
- 新增或修改测试模板。
- 修改测试参数含义。
- 修改轮数、并发、默认 Prompt。
- 修改报告字段。
- 修改页面提示卡、说明文案或默认操作路径。
- 修改日志或报告保存位置。
- 修改 token 消耗逻辑。
- 修改进度、取消、任务机制。
- 修改启动方式或端口策略。

原则：功能可以变，但使用者不能靠猜。每次功能变更后，手册必须保持可用、准确、同步。
